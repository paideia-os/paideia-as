## 1. Tempo + cross-repo handoff

v0.8 runs continuously across all selected V-issues with **five named pause-and-verify checkpoints (A–E)** plus a round-close ritual (m7). This is a refinement of the PA7-completion two-checkpoint cadence: v0.8 has one checkpoint per V-group milestone (m1–m5), because the 9 remaining quarantined files split across the five gap-families.

Per memories `feedback_cross_repo_escalation`, `feedback_phase6_to_paideia_os_resume`, and `feedback_paideia_as_version_discipline`, the checkpoint pauses are existing protocol. This round *extends* the unquarantine-verify-resume cadence so that **each V-group milestone close ends with a paideia-os unquarantine attempt**, not only at the round close.

### 1.1 Per-issue cycle (between checkpoints)

Per the user's directive — "softarch → workerbee → debugger per issue":

```
softarch  → produces the issue body (v0.8 template, §5)
              + commits to the GitHub issue
              + dispatches workerbee
workerbee → implements the V-item
              + runs cargo test --workspace
              + runs the cross-repo canaries (§6.1)
              + commits + pushes if green and test count grew
debugger  → invoked ONLY when:
              (a) cargo red on workerbee's branch, OR
              (b) test count flat or dropped, OR
              (c) SARIF golden mismatch after intentional catalog change, OR
              (d) cross-repo canary red
            (no debugger invocation on green issues)

if cargo green AND test count strictly grew AND canaries green:
  commit + push to main         (PaideiaOS-mode no-PR workflow, per Phase 4 §3)
  gh issue close <n>
  → immediately pick up next issue; no pause unless milestone-close-triggered
```

The debugger is *not* invoked speculatively. softarch composes the issue, workerbee executes it, debugger only runs on red. This is the canonical three-agent rotation called out by the user directive; documenting it here makes it a first-class round process.

### 1.2 Checkpoint A — m1 close (V1 + V8)

**Trigger:** V1 (st_value regression) + V8 close.

**Rationale:** V1 fixes the symbol `st_value` regression introduced by PA7-completion G1. paideia-os was last verified clean at `dfac617` (checkpoint-1 unquarantine commit). Bumping the submodule pin past that commit may regress that build, so checkpoint A is a *correctness regression verification* — does paideia-os still build at the pinned commit after V1 lands?

**Verification ritual:**

```bash
# In paideia-as (no-op; already green):
cargo test --workspace
# Verify the st_value regression test names the exact byte:
cargo test --workspace -- regression::st_value

# In paideia-os (the cross-repo verify):

git submodule update --remote tools/paideia-as
./tools/build.sh

# If clean, commit the submodule bump.
git add tools/paideia-as
git commit -m "$(cat <<'EOF'
Bump paideia-as submodule (m1 close — V1 st_value regression + V8)

V1 fixed the symbol export st_value math regression introduced by
PA7-completion G1. Build verified: ./tools/build.sh produces kernel.elf
cleanly against the new pin; the dfac617-era unquarantined files
still resolve.

Co-Authored-By: workerbee <noreply@anthropic.com>
EOF
)"
git push origin main
```

If `./tools/build.sh` is red after the bump, the failure mode is a *post-V1 regression in V1 itself* — file a new paideia-as issue under `pa8-m1-foundational`, label `regression-verify`, and resume the paideia-as autonomous loop on it. Do not bump the submodule pin yet.

## 2. PR sizing discipline

v0.8 keeps the PA7-completion size bands but reserves **L** for **V5 only** — the width-threading sweep that touches 11 Mov sites. Every other V-item must split if it crosses 400 LOC net diff.

### 2.1 Size bands (this round)

| Size | Net diff (LOC, ex. generated, ex. corpus, ex. snapshots) | Examples | Files touched | Review target |
|---|---|---|---|---|
| **XS** | < 50 | V1 st_value regression fix (single math change), V8 small cleanup, single-mnemonic addition in V10 | ≤ 3 | ≤ 10 min |
| **S** | 50–200 | V2 / V3 / V4 single elaborator gap, V6 cast-width matcher, V7 sub-register slice, V11 single memory operand form | ≤ 6 | ≤ 25 min |
| **M** | 200–400 | V9 dead-arm activation (multiple sites if cohesive), V11 general memory operand form | ≤ 12 | ≤ 45 min |
| **L** | 400–800 | **V5 width-threading sweep ONLY** if it stays cohesive; if it grows past 200 LOC, split into one PR per Mov site (11 PRs) | ≤ 20 | ≤ 90 min |

Generated code, test corpora, and `Cargo.lock` lines are excluded from the LOC count. SARIF snapshot regen output is excluded.

### 2.2 V5 sub-split rule

V5 (width-threading across 11 Mov sites) is the only L candidate. The rule:

- If V5 stays under 200 LOC net after the framework + one-site change, keep it as a single L PR.
- If V5 crosses 200 LOC, **split into one issue per Mov site** with shared framework change as V5.0 (foundation), then V5.1 through V5.11 each landing one site.
- Each sub-issue then sizes as XS or S; no sub-issue may carry the L tag.

This is the explicit anti-bloat rule for V5; analogous to Phase 5's `area:walker-activation` chain.

### 2.3 Hard-split triggers (this round)

A single V-item must split if any of:

1. Net diff projects above 400 LOC (V5 excepted per §2.2).
2. It mixes two gap-families (e.g., V2 elaborator + V11 encoder bridge).
3. It activates more than one structural arm cohesively unrelated to the others (V9 sub-split allowed if needed).
4. It mixes a `correctness-sweep` change with an `activate-dead-code` change (different `gap:*` label, different test surface).
5. It mixes V1–V11 mandatory work with V15 cleanup work (different label triad, different review priority).

### 2.4 No-PR mode reminder

Per Phase 4 retrospective §3 (carried through Phase 5, Phase 6, PA7-completion): under PaideiaOS-mode, work lands as direct pushes to `main` after `cargo test --workspace` green. The size bands still apply — the "PR" is the squash-merge commit. A single commit exceeding the band ceiling is forbidden the same way an XL PR would be in PR mode.

## 4. Milestone shape

v0.8 introduces **one milestone per V-group**, not a single round-wide milestone. This is a refinement of the PA7-completion single-milestone shape, because the five V-groups gate cleanly on five distinct unquarantine checkpoints.

### 4.1 Naming + role

| Milestone | Title | Owns V-items | Close trigger |
|---|---|---|---|
| `pa8-m1-foundational` | v0.8 m1 — Foundational (regression + small cleanup) | V1, V8 | both closed + checkpoint A verified (§1.2) |
| `pa8-m2-elaborator-gaps` | v0.8 m2 — Elaborator gaps | V2, V3, V4 | all three closed + checkpoint B unquarantine batch verified (§1.3) |
| `pa8-m3-correctness-sweep` | v0.8 m3 — Correctness sweep (widths + casts + sub-regs) | V5, V6, V7 | all three closed + checkpoint C unquarantine batch verified (§1.4) |
| `pa8-m4-dead-code-activation` | v0.8 m4 — Dead-code activation | V9 | closed + checkpoint D activated-arm tests green (§1.5) |
| `pa8-m5-mnemonic-bridge` | v0.8 m5 — Mnemonic bridge (supervisor ops + memory operand general form) | V10, V11 | both closed + checkpoint E final unquarantine batch verified (§1.6) |
| `pa8-m6-cleanup` | v0.8 m6 — Cleanup | V15 (and V16 if scope B is widened) | all closed; no cross-repo checkpoint |
| `pa8-m7-closure` | v0.8 m7 — Closure (smoke + unquarantine sweep + submodule bump + tag) | (no V-items; ritual-only) | §9.3 closure ritual complete |

m1 through m5 each gate a cross-repo checkpoint (§1.2 through §1.6). m6 has no cross-repo gate — it is paideia-as-internal cleanup. m7 is the round-close ritual; it has no new V-items, only the closure-day work.

### 4.2 Within-milestone sequencing

osarch owns the within-milestone order. The softarch invariants the order must respect:

1. **m1**: V1 (regression) before V8 (cleanup). V1 ahead because checkpoint A verification depends on V1's fix landing.
2. **m2**: V2, V3, V4 may land in any order; checkpoint B runs once all three close.
3. **m3**: V5 (width sweep) lands first; V6 (cast-width) second; V7 (sub-register) third. The V5 sub-split (§2.2), if triggered, lands all V5.* before V6 begins.
4. **m4**: V9 is single-issue (sub-split allowed if §2.3 trigger 3 fires).
5. **m5**: V10 (mnemonics) before V11 (memory operand). V10 ahead because some V11 forms reference V10 mnemonics.
6. **m6**: V15 standalone.
7. **m7**: no V-items; only the ritual.

### 4.3 Round-close criterion (the bounded stop point)

The autonomous loop stops *only* when all of:

| # | Criterion | How verified |
|---|---|---|
| 1 | All scope-B V-issues closed (V1–V11 + V15). | `gh issue list --label pa8 --state open` empty (or only V12–V14/V16 deferred). |
| 2 | All 9 paideia-os quarantined files unquarantined and on `main`. | `cd ../PaideiaOS && test -z "$(find .quarantine -type f 2>/dev/null)"`. |
| 3 | paideia-os `./tools/build.sh` produces a non-empty `kernel.elf` against the v0.8.0 submodule pin. | `cd ../PaideiaOS && ./tools/build.sh && test -s build/kernel.elf`. |
| 4 | `paideia-as --version` reports `0.8.0`. | `paideia-as --version \| grep -q '^paideia-as 0.8.0$'`. |
| 5 | Tag `v0.8.0` pushed. | `git tag --list v0.8.0 && git ls-remote --tags origin v0.8.0`. |
| 6 | `CHANGELOG.md` `## v0.8.0 — Structural correctness` section landed. | `grep -q '^## v0.8.0' CHANGELOG.md`. |
| 7 | `design/toolchain/phase-transition-8.md` retrospective written. | File exists; references each per-issue design note. |
| 8 | paideia-os submodule pin bumped to v0.8.0 tag and pushed. | `cd ../PaideiaOS && git submodule status tools/paideia-as | head -1 | awk '{print $1}'` matches v0.8.0 SHA. |

V12, V13, V14, V16 (the deferred set) are *not* gating: they re-classify to `phase:9` and the round closes.

### 4.4 Scope-B widening rule

If osarch (per its companion) recommends widening scope B to include V12 / V13 / V14 / V16 mid-round, the widening lands as:

1. A new label combination on the widened V-items (`pa8` + `gap:anticipated` + new V-group label).
2. A new milestone `pa8-m6-cleanup` slot (V15 already there; V16 joins it; V12/V13/V14 spawn `pa8-m6-extended` if needed).
3. No retroactive renaming of m1–m5 closures already taken.
4. §4.3 criteria do not change; widening adds work but does not redefine the stop.

The default in this plan is **scope B = V1–V11 + V15 (12 items)**. Wider scope is opt-in via the osarch companion.

| V11 | New memory operand forms may need new T-codes for malformed operand diagnostics. | likely T-code |

### 7.2 V9 catalog-pre-flight check (NEW THIS ROUND)

Before V9 lands, the workerbee must:

1. Read each of T0520, T0526, T0527, T0528 from `catalog.toml`.
2. Confirm the text is grammatical, complete, and points users to a documented remediation.
3. Confirm the SARIF properties block (severity, kind, help.text) is correctly populated.
4. If any of (1)–(3) fail, the catalog fix lands as a separate `[pa8-m4-pre]` commit before V9's activation.

This is unique to V9 because the codes were authored in past rounds with the assumption they would fire later. v0.8 is "later".

### 7.3 Regen ritual (per memory `feedback_paideia_as_version_discipline`)


The single-commit rule is load-bearing: a `.snap.new` file left in the tree is a process violation that surfaces as a CI red on the next PR; it is cheaper to catch in-commit. (paideia-as does run CI; only paideia-os does not — memory `feedback_paideia_os_no_cicd`.)

## 8. Unquarantine protocol

Same 5-step protocol from PA7-completion §8, applied at each of checkpoints A–E. The protocol is unchanged; the cadence is what v0.8 changes (five checkpoints instead of two).

### 8.1 Per-file protocol (5 steps)

```bash
# Step 1: Move the file back from .quarantine/ to its real location.
cd /home/snunez/Development/PaideiaOS
git mv .quarantine/src/kernel/PATH src/kernel/PATH

# Step 2: Single-file paideia-as check (fast path — exits 0 if the unquarantined
# file's surface compiles cleanly against the bumped submodule pin).
$(./tools/find-paideia-as.sh) check src/kernel/PATH

# Step 3: Full kernel build — must produce kernel.elf cleanly.
./tools/build.sh

# Step 4: Commit the per-file unquarantine (or per-checkpoint batch — see §8.2).
git add src/kernel/PATH .quarantine/
git commit -m "$(cat <<'EOF'
unquarantine: src/kernel/PATH

Re-enabled after paideia-as v0.8 #<issue> closed.
Build verified: ./tools/build.sh produces kernel.elf cleanly.

Co-Authored-By: workerbee <noreply@anthropic.com>
EOF
)"

# Step 5: If the file trips a different paideia-as gap (not the V-item just
# closed), revert the mv, document the new gap, and file a follow-up v0.9
# issue. Do NOT leave the file half-moved.
# (Revert: git checkout HEAD -- src/kernel/PATH .quarantine/)
```

Five steps total. Step 1 is the move; step 2 is the per-file check; step 3 is the full build; step 4 is the commit; step 5 is the fallback (revert + document + file follow-up).

### 8.2 Per-batch vs per-file commit choice

Default: per-checkpoint batch (one commit per checkpoint, cleaner history, matches §1.2–§1.6 verification). Fallback to per-file commits when a batch is red and bisecting is needed.

### 8.3 Batch commit-message template

```
unquarantine: <checkpoint name> batch (<N> files)

Re-enabled after paideia-as v0.8 #<V-issues> closed.
Build verified: ./tools/build.sh produces kernel.elf cleanly.

Files:
  - src/kernel/core/ipc/slots.pdx
  - src/kernel/core/ipc/allocator.pdx
  - <...>

Co-Authored-By: workerbee <noreply@anthropic.com>
```

### 8.4 Rollback procedure

If step 3 (`./tools/build.sh`) is red: do not commit; `git checkout HEAD -- src/kernel/PATH .quarantine/` to restore both sides (the `git mv` is fully reversible because `.quarantine/` is preserved); file a new paideia-as issue under the appropriate `pa8-m{N}` milestone with the build error attached (label `gap:byte-emit` or `gap:parser-surface`); resume the paideia-as autonomous loop on the new issue; re-try the unquarantine after it closes.

If the new gap is **not** in v0.8 scope (i.e., it surfaces a v0.9 concern), file the follow-up against `phase:9` label and leave the file in `.quarantine/` until v0.9. This is the explicit escape valve.

## 9. Closure rituals

Three rituals: per-issue, per-checkpoint, round-close.

### 9.1 Per-issue ritual

```
commit + push to paideia-as main         (PaideiaOS-mode no-PR workflow)

```

No design-doc requirement per-issue beyond the `## Notes` field in the issue body and any inline `design/toolchain/<topic>.md` note the issue's AC names. The roll-up is at round close (§9.3).

### 9.2 Per-checkpoint ritual (five times in the round)

Each of checkpoints A–E follows the same shape:

1. paideia-as: confirm cargo green, test count grew, all milestone V-issues closed.

5. paideia-os: push to main.
6. paideia-as: resume autonomous loop on the next milestone's V-items.

Checkpoint A is unique: it only bumps the submodule pin (no unquarantine batch), because m1 verifies a regression-fix, not a new unblock. Checkpoints B–E each unquarantine a batch.

### 10.1 The 5-step protocol

```bash
# Step 1: paideia-os work hits a paideia-as gap. The failure is observed as
# either a build failure of an unquarantined file or a runtime correctness
# regression in a downstream kernel test.

# Step 2: File the paideia-as issue with the cross-repo escalation labels.
cd /home/snunez/Development/paideia-as
gh issue create \
  --label "cross-repo:paideia-as-blocked,pa8,v0.8,gap:byte-emit,gated:downstream-paideia-os" \
  --title "[<area>] <imperative summary>" \
  --body "$(cat <<'EOF'
## Surfaced by
paideia-os@<sha>

## Linked paideia-os issue
paideia-os#<n>

## Summary
<what the paideia-os work was attempting; what failed; what byte/symbol/code is wrong>

## Definition of done
<the observable paideia-as test that fails before this change and passes after>
EOF
)"

# Step 3: Run the three-agent loop on the new paideia-as issue.
#         softarch → workerbee → debugger (per §1.1).
#         Land the fix as a direct push to main with the §5.1 commit message form.

# Step 4: Close the paideia-as issue. Bump the paideia-os submodule pin to
# point at the fix commit.
cd /home/snunez/Development/PaideiaOS
git submodule update --remote tools/paideia-as
git add tools/paideia-as
git commit -m "Bump paideia-as submodule (fix for #<paideia-os-issue>)"
git push origin main

# Step 5: Resume paideia-os autonomous loop on the original issue, which is
# now unblocked.
```

Five steps total. The protocol is symmetric across rounds; v0.8 inherits it without modification. The only round-specific binding is the label set (`pa8` + `v0.8` + the appropriate `gap:*` label).

If the cross-repo escalation surfaces a gap that does not fit v0.8 (e.g., a v0.9-class architectural change), file the paideia-as issue with `phase:9` instead of `pa8` and leave the offending paideia-os file in `.quarantine/`. The paideia-os work resumes around the gap, not through it.

### 10.3 The standing pattern

The four-round cohort (Phase 5, Phase 6, PA7-completion, v0.8) has produced the same shape of cross-repo handshake four times in a row. Future rounds (v0.9+) MUST inherit this protocol verbatim. The protocol is no longer round-specific; it is the standing pattern for paideia-as ↔ paideia-os development.
