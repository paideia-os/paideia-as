# PaideiaOS x86_64 Calling Convention (paideia-as focus)

**Status:** Specification v1.0 (Phase 2)  
**Date:** 2026-06-18  
**Scope:** Canonical register discipline and calling convention for PaideiaOS native code (paideia-as). Complements `abi.md` (which covers ELF metadata and versioning).

---

## Overview

This document specifies the calling convention for PaideiaOS native code — applications, drivers, and userspace servers written in paideia-as. The System V AMD64 ABI is used only at specific boundaries (ACPICA, WASM jail); the native convention is optimized for PaideiaOS' effect handlers and capability-typed registers.

The convention is a property of the paideia-as elaborator and emitter; both enforce it via type and effect inference.

---

## 1. Register file partitioning

The 16 x86_64 general-purpose registers are partitioned into five bands:

| Band | Registers | Discipline | Purpose | Saved |
|---|---|---|---|---|
| **General** | RAX, RBX, R8–R11 | unrestricted | Scratch / general computation | RAX, RCX, RDX, R8–R11 caller-saved; RBX callee-saved |
| **Capability** | R12, R13 | linear; LAM-tagged | Passing capability handles in/out | Callee-saved |
| **Effect** | R14, R15 | affine | Handler stack and environment pointers | Callee-saved |
| **Reserved** | RSP, RBP, RIP | reserved | Stack, frame, instruction pointers | Architectural |
| **Argument** | RDI, RSI, RDX, RCX | unrestricted | Argument passing (integer/pointer) | Caller-saved |

The vector register file (ZMM):
- **ZMM0–ZMM15:** scratch / general SIMD (caller-saved).
- **ZMM16–ZMM31:** reserved for vectorized PQ crypto state (callee-saved; a non-PQ function preserves them across calls but a PQ kernel may use freely).

## 2. R15 / AS3 resolution (Effect environment pointer)

### 2.1 The role of R15

Per the PaideiaOS design (custom-assembler.md §8.4), **R15 holds the active effect-handler environment pointer**. The environment is a table of installed handlers; effect dispatch looks up a handler in this table in two instructions:

```asm
mov rax, [r15 + handler_offset]  ; offset is compile-time constant
call rax
```

### 2.2 Handler-bracketed regions (prologue/epilogue)

When entering a `handle e { ... }` block:

1. **Prologue:** Save the outer environment pointer (R15) on the stack.
2. **Move:** Point R15 at the new handler environment.
3. **Body:** Execute the handler-bracketed code.
4. **Epilogue:** Restore the saved R15 (pop from stack).

This is implemented via `emit_handler_open()` and `emit_handler_close()` in the emitter.

**Byte sequences:**

```asm
; emit_handler_open(buf, new_env_reg):
push r15            ; save outer environment pointer (2 bytes: 41 57)
mov r15, new_env_reg  ; point R15 at new handler env (3 bytes: REX + 89 + ModR/M)

; emit_handler_close(buf):
pop r15             ; restore outer environment (2 bytes: 41 5f)
```

### 2.3 System V bridge (C interop, AS3 resolved)

When PaideiaOS code calls out to C (System V ABI), R15 must be saved and cleared because:
- The System V ABI assumes R15 is callee-saved and unused by C code.
- The C function has no concept of PaideiaOS effect handlers.
- On return, R15 must be restored.

The bridge thunk (generated by the macro library) wraps the C call:

```asm
; emit_sysv_bridge_prologue(buf):
push r15            ; save effect environment (2 bytes: 41 57)
                    ; (optional: clear R15 = xor r15, r15 if needed)

; ... call out to C ...

; emit_sysv_bridge_epilogue(buf):
pop r15             ; restore effect environment (2 bytes: 41 5f)
```

This resolves **AS3** from custom-assembler.md §15: "The effect-environment pointer in R15 conflicts with the System V ABI assumption that R15 is callee-saved. The thunk-bridge logic must save R15 on entry to C and restore on exit."

### 2.4 Nested handlers (Phase 2 minimum)

When one handler installs inside another (nested effects), the new handler's environment record points to the saved outer R15 via a `.parent` field. The `emit_handler_chain()` function emits the open sequence; link-stage constructs the parent chain. Phase-2-m11 minimum: same bytes as `emit_handler_open()`, with the `.parent` field populated at link time.

## 3. Argument passing

### 3.1 Integer / pointer arguments
- **Positions 1–4:** RDI, RSI, RDX, RCX (in declaration order).
- **Position 5+:** stack, 8-byte aligned, in declaration order.

### 3.2 Capability arguments
- **Position 1:** R12 (linear, LAM-tagged).
- **Position 2:** R13 (linear, LAM-tagged).
- **Position 3+:** stack with capability-binding metadata (consumed by paideia-link).

### 3.3 Floating-point / vector arguments
- **Positions 1–4:** ZMM0–ZMM3 (in declaration order).
- **Position 5+:** stack, 64-byte aligned, in declaration order.

### 3.4 Return values

| Type | Register |
|---|---|
| Integer / pointer | RAX |
| Capability | R12 |
| Floating-point / vector | ZMM0 |
| Tuple (fits in registers) | RAX + RDX (int); RAX + R12 (mixed); ZMM0 + ZMM1 (float) |
| Large struct (> 16 bytes) | Caller provides return pointer in RDI; callee writes to it; returns pointer in RAX |

---

## 4. Callee-saved registers (function prologue/epilogue)

A function must preserve (save and restore) these registers:
- RBX, RBP, R12, R13, R14, R15, RSP.
- ZMM16–ZMM31.

The prologue/epilogue generator (`emit_prologue()`, `emit_epilogue()` in prologue.rs) handles this automatically when registers are listed in `FrameLayout::saved_regs`.

**If R15 must be saved:** include it in the register list, and the emitter will emit `mov [rbp - offset], r15` and `mov r15, [rbp - offset]` using the same slots as other callee-saved registers.

## 5. Stack discipline

### 5.1 Alignment

- **PaideiaOS native:** Stack is **64-byte aligned** at every call boundary (versus System V's 16-byte).
- **Rationale:** Accommodates ZMM register saves without unaligned-access penalties.

### 5.2 Red zone

- **PaideiaOS:** **No red zone** (0 bytes).
- **Rationale:** Exception/interrupt entry would corrupt a red zone on PaideiaOS frames.
- **Contrast:** System V AMD64 ABI permits a 128-byte red zone below RSP.

### 5.3 Frame layout

```
   high address
   ┌─────────────────────────────────┐
   │ Caller's arguments on stack     │  (positions 5+)
   ├─────────────────────────────────┤
   │ Return address                  │  pushed by CALL
   ├─────────────────────────────────┤  ← RSP at entry
   │ Saved RBP (if frame-pointer)    │
   ├─────────────────────────────────┤  ← RBP
   │ Saved callee-saved registers    │
   │   RBX, R12, R13, R14, R15       │
   │   ZMM16..ZMM31 (if used)        │
   ├─────────────────────────────────┤
   │ Local variables                 │
   ├─────────────────────────────────┤
   │ Outgoing argument area          │
   ├─────────────────────────────────┤  ← RSP, 64-byte aligned before call
   │ (no red zone)                   │
   ▼  low address
```

## 6. Capability discipline

R12 and R13 carry LAM-tagged 64-bit capability handles:
- High 15 bits: tag bits (enforced by hardware LAM feature).
- Low 49 bits: pointer value.

**Linearity enforcement:** The paideia-as type system enforces linear consumption of capabilities on entry/exit. A caller does not need to save R12/R13 across calls because they are callee-saved.

---

## 7. When to save R15

R15 is callee-saved, meaning a function must preserve it unless:
- The function has no effect-handler installation in its body.
- The function's effect signature is identical to the caller's (no environment switch).

In practice:
- **Leaf functions** (no calls to effect-typed functions) do not need to save R15.
- **Functions that call functions with different effect signatures** save R15 via the handler-bracketed region prologue/epilogue.
- **System V bridge thunks** save R15 unconditionally (via `emit_sysv_bridge_prologue()` and `emit_sysv_bridge_epilogue()`).

## 8. Summary: R15 save/restore patterns

| Scenario | Where saved | Byte sequence | References |
|---|---|---|---|
| **Handler-bracketed region** | Stack (pushed) | `41 57` (push r15) | `handler.rs::emit_handler_open()` |
| **Handler-bracketed region exit** | Stack (popped) | `41 5f` (pop r15) | `handler.rs::emit_handler_close()` |
| **System V bridge entry** | Stack (pushed) | `41 57` (push r15) | `sysv_bridge.rs::emit_sysv_bridge_prologue()` |
| **System V bridge exit** | Stack (popped) | `41 5f` (pop r15) | `sysv_bridge.rs::emit_sysv_bridge_epilogue()` |
| **Function prologue (if R15 in saved_regs)** | Stack (mov to [rbp-offset]) | `4C 89 7D F0` (mov [rbp-16], r15) | `prologue.rs::emit_prologue()` |
| **Function epilogue (if R15 in saved_regs)** | Stack (mov from [rbp-offset]) | `4C 8B 7D F0` (mov r15, [rbp-16]) | `prologue.rs::emit_epilogue()` |
