## File 1: `core/cap/slab.pdx`

**Surface kind:** Real .pdx code (functions + module state)

**File type:** Capability slab allocator with `slab_alloc()` and `slab_free()` functions.

**`check` outcome:** Exit 1  
Error P0158: "block expression must have a final expression; trailing `;` is not allowed"
- Location: line 112, column 32 (if block spanning 4 lines)

**`build` outcome:** Fails at same point.

**Specific blocker:**  
The `slab_free` function (lines 111–116) has an `if` block without a final expression:
```
let slab_free : (u64) -> () = fn (idx : u64) -> {
  if idx < CAP_SLAB_CAPACITY {
    free_list[idx] = free_head;
    free_head = idx;
  }
}
```
The `if` lacks an `else` branch and the trailing statements inside use `;`, making it an incomplete block expression. In paideia-as, an `if` without `else` must either be in statement position (not return a value) or have both branches return a value. Here, the block as a whole is being returned from the function, but the `if` gives no value to the final expression.

**Diagnosis:** Unit-return functions cannot use bare `if` with statements. Needs either:
- Wrap the `if` in a statement position (use `{}` or remove semicolons), or
- Add an explicit `else` clause returning `()`.

**Suggested v0.9 issue title:**  
`P0158 elaboration: bare if in unit-return fn needs explicit else or statement context`

**Surface kind:** Pseudocode stub (comment-based, bare NAME = VALUE)

**File type:** Constants and comments only; no executable code.

**`check` outcome:** Exit 1  
Multiple E0011 (reserved-word used as identifier) errors from Unicode `§` and `×` characters in comments.  
Error E0006 (invalid number literal) on "-001" in "ipc-slots-001.md".  
Error P0100 (expected item) on line 1; parser cannot recognize `;` as part of the module syntax.

**`build` outcome:** Same parse errors.

**Specific blocker:**  
The file is pure pseudocode:
```

...

...

```
Paideia-as parser expects `module Slots = { ... }` (with `=` and `{}`), not the bare `module ... end module` syntax. Additionally, bare comments starting with `;` are not valid at the top level; they need to be inside a module body or marked as documentation.

**Diagnosis:** Pseudocode file using legacy/non-existent syntax (`;` comments at top level, bare `module ... end module`). Not real paideia-as code.

**Suggested v0.9 issue title:**  
`Rewrite pseudocode stub: slots.pdx to real .pdx with module = structure { ... }`

## File 7: `core/ipc/channel.pdx`

**Surface kind:** Real .pdx code (structs + module state + array initializer)

**File type:** Channel and Message structs, global channel pools, constants.

**`check` outcome:** Exit 0 (no errors from check; file parses successfully)

**`build` outcome:** Exit 1  
Error P0211 (array repeat count must be constant literal integer) on lines 60 and 61.  
Error P0101 (expected statement, found empty statement `;`) on line 61.  
Error P0100 (expected item) on line 62.

**Specific blocker:**  
The file tries to initialize a large array of `Channel` struct instances:
```rust
let mut channel_pools : [[Channel; 32]; 1] = [
  [
    Channel { producer_cap: 0, consumer_cap: 0, head: 0, tail: 0, slots: [Message { length: 0, payload: [0; 256] }; 64] };
    Channel { ... };
    ...
  ]
]
```

The problem is twofold:
1. **Nested array repeat:** `[0; 256]` inside a struct initializer that is itself repeated via `;` syntax is parsed as needing a constant repeat count. The inner `256` and the outer `64` must be literal integers, which they are, but the elaborator is deferring validation (P0211).
2. **Semicolon syntax in array literal:** The use of `;` to separate repeated struct instances in an array is ambiguous in the parser context — it looks like a statement terminator, not an array separator.

This is a **real code file with a legitimate elaborator gap**: the encoder does not yet support nested array repeat counts in struct initializers within mutable global declarations.

**Diagnosis:** Elaborator gap: nested array repeat counts in struct field initializers (e.g., `payload: [0; 256]` inside a Channel struct that is itself repeated) are deferred. The parser flags P0211 but the encoder would fail even if the parser passed.

**Suggested v0.9 issue title:**  
`P0211: Array repeat counts in nested initializers (struct field arrays repeated via ; syntax)`

## File 9: `core/sched/enqueue.pdx`

**Surface kind:** Real .pdx code (functions, mutable state, control flow)

**File type:** Runqueue enqueue/dequeue with priority bitmap.

**`check` outcome:** Exit 0 (successful)  
Debug output: "[lower_ast_to_ir] Lambda AST node X -> IR X" for 5 nodes (no errors).

**`build` outcome:** Exit 2  
Error: "encoder failed on IR node 7: encoding error: Unsupported("mov form not in phase-3-m2-002 minimum")"

**Specific blocker:**  
The encoder rejects a `mov` instruction form that is not yet in the phase-3-m2-002 minimum instruction set. Looking at the code, the issue is likely:
- Line 60: `let head : u64 = heads[priority];` — array indexing via register (needs base+index form)
- Line 63: `heads[priority] = tcb;` — array write via register indexing

The elaborator successfully lowers the array access to IR, but the encoder (paideia-as-emitter-elf) does not yet support the register-base addressing mode required for `heads[priority]` (where `priority` is a register, not a constant).

**Diagnosis:** Encoder gap: `mov` with array indexing via register operand (e.g., `mov [rax + rbx*8], rcx`) is not in the v0.8.0 instruction whitelist (phase-3-m2-002 minimum). The code is semantically correct but requires the base+index+scale addressing mode, which is planned for a future phase.

**Suggested v0.9 issue title:**  
`P3-m2-003: mov form not in phase-3-m2-002 minimum — array indexing via register base+index addressing`

## Recommendations for v0.9 Roadmap

1. **Pseudocode rewrites (5 files):** Convert slots, allocator, dispatch, mpsc_lock, destroy_channel from bare comments to real paideia-as code with `module = structure { ... }` syntax and proper let-bindings.

2. **P0158 (slab.pdx):** Emit a better error message guiding the user to add an `else` clause or restructure as a statement. Alternatively, detect bare `if` in block-expression position and offer to insert the missing `else ()`.

3. **P0211 (channel.pdx):** Extend the elaborator to support nested array repeat counts in struct field initializers. This is a known gap for initializing large global data structures.

4. **Encoder addressing modes (enqueue.pdx):** Add support for base+index register addressing (`mov [reg+reg*scale], reg`) to the instruction encoder. This is currently a known limitation for phase-3-m2-002 (minimum).

5. **Unquarantine tlb_shootdown.pdx:** This file is ready; move to main tree or file a separate issue for MMIO/INVLPG encoder support if needed.
