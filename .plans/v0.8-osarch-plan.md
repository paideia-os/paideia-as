## 0. Scope decision

Three scope options are available:

- **(A) Minimum — V1 verify regression + V2 if-as-final-expression + V3 array-literal initialisers + V4 pointer-operand binding.** Closes only the four reported elaborator/encoder gaps that block individual quarantined files. Round size ~8–10 issues. Rejected: it leaves V5/V6/V7 latent (sub-register, width threading, cast lowering) — the slab/IPC/sched files would unquarantine but their codegen would silently be wrong (8x wider, wrong cast widening, wrong opcode family for sub-register `mov al, imm8`). The PaideiaOS R-phase resume would inherit a kernel.elf that links but mis-encodes. A second integration sweep would be unavoidable inside R-phase work.
- **(B) Recommended — V1 through V11, plus V15 (debug trace cleanup) and V16 (peer Mov sweep, which is the same work as V5 framed as cleanup).** Includes V1 regression verification, the three reported elaborator gaps (V2-V4), the encoder correctness sweeps (V5 width threading, V6 cast lowering, V7 sub-register encoding), the structural-correctness floor (V8 symbol `st_value`, V9 activate dead-code arms), the supervisor-surface bridge (V10) and the general memory-operand form (V11). Round size ~24 issues. **Picked.**
- **(C) Full — V1 through V16, incorporating V13 (`.data` array literals — same as V3 framed differently), V14 (real string literal surface), V15 multiboot2 / PVH Note.** Adds two PaideiaOS-quality-of-life items not blocking R-phase reactivation. Round size ~28–32 issues. Rejected for v0.8 because (a) V14 strings can be worked around with the existing `[u8; N]` byte-array shape that boot/banner.pdx already uses, and (b) V15 multiboot2 / PVH is gated by the `qemu -kernel -device loader,addr=0x100000` workaround that has been stable across v0.6.0 + v0.7.0; neither is on the checkpoint-2 unquarantine critical path. Both move to v0.9.

**Decision: option (B).** Justification:

1. **(B) is the smallest scope that drains the entire `.quarantine/` directory.** Stopping at (A) unquarantines individual files but ships a kernel.elf whose `.text` is wrong at the instruction level (V5/V6/V7) — discovered the moment the kernel is exercised. The cost of (B) over (A) is ~14 issues, mostly XS/S, mostly in the encoder where Phase 6 left the infrastructure already in place.
2. **The correctness sweeps (V5/V6/V7) are forced moves.** They are not opportunistic improvements — they fix outright wrong codegen that the PA7-completion round shipped because the m4 work landed in a single emit-walker site, leaving the other eleven sites and the corresponding cast/sub-register paths untouched. Every quarantined file uses at least one of these.
3. **V8 (real `st_value`) and V9 (lower.rs Unsafe activation) are structural-floor work that PA7C deferred.** PA7C-m1-001 made symbols export with real names; PA7C-m2-001/002/003 added the structural reception arms for `RawInstruction`, scratch-let, and `Operand::Var`. Both stopped one step short of working end-to-end: the symbols have wrong `st_value`, and the structural arms are dead code because `lower.rs` does not emit `RawInstruction` from `ExprData::Unsafe`. Closing them now lets the PA7C-shipped infrastructure earn its keep.
4. **V10 + V11 unblock the next round (v0.9 / PaideiaOS Phase 7 / R7-driver work) for free.** The supervisor mnemonics already have encoders; the missing bridge is one match-arm fan-out (V10). The memory operand general form is needed by the channel/IPC files' descriptor field reads (V11) — without it, `(*ch).head` reads emit placeholder bytes even after V4 closes their *write* side. Deferring V10/V11 to v0.9 forces a second sweep against the same files.
5. **G14 (strings) and G15 (multiboot2 / PVH) defer cleanly to v0.9.** Verified: the only string use in any quarantined file is comment text; banner.pdx already lives outside the quarantine and uses `[u8; 64]`. The `-device loader,addr=0x100000` QEMU command line in `tools/run-qemu.sh` is unchanged across the last three rounds. Neither blocks R-phase reactivation.

**Rejected alternative:** option (A) with V5/V6/V7 deferred to a separate "encoder correctness" round before R7. Rejected because that scheme creates two cross-repo integration sweeps in the same calendar window for the same files: the (A) unquarantine sweep, then the correctness-sweep follow-up. The cost of bundling them into v0.8 is ~14 issues; the cost of splitting is at least two more PaideiaOS submodule pin bumps and the associated R-phase pause/resume cycles per `feedback_paideia_os_tempo.md` and `feedback_phase6_to_paideia_os_resume.md`.

- **V13 (data-section array initialisers as a separate item).** V3 already covers this — the IR shape `IrKind::ArrayLit { elems }` and the `.data` blob lowering are the same work whether named V3 (elaborator gap) or V13 (G13 PaideiaOS QoL). Issue m2-003 implements it once.
- **V14 (real string-literal surface).** Deferred to v0.9. The `"banner text\n"` form does not parse today; the workaround is `[u8; N]` byte arrays with explicit element literals. No quarantined file uses string literals.
- **V15 (multiboot2 / PVH ELF Note).** Deferred to v0.9. The `-device loader,addr=0x100000` workaround is stable; QEMU loads the ELF and jumps to its entry point.
- **V16 (peer Mov sweep) as a separate milestone.** Folded into V5 (m3-001); the eleven Mov sites enumerated in the gap inventory are the explicit acceptance criteria of V5.
- **Self-hosting work.** Tier 2/3 self-host crate ports remain Phase 8+/v0.9+. v0.8 stays in the cross-compile build path.
- **paideia-as-linker integration.** PaideiaOS continues to invoke GNU `ld` via `tools/build.sh`.
- **Multi-file build orchestration.** `paideia-as build a.pdx b.pdx -o linked.elf` stays deferred; per-file objects + `ld` glue continue.
- **DWARF emission.** `paideia-as-dwarf` crate exists; wiring into `--emit elf64` stays Phase 8+.
- **Optimisation passes in the build path.** Peephole / DSE / encode-tight / unroll remain side-table-level.
- **Atomic ops + LL/SC primitives.** R5.5/R6.5 used `mut` globals with single-CPU serial assumption; SMP-safe atomics land when bring-up begins. The `mpsc_lock.pdx` file uses constants only — it does not need `lock cmpxchg` to unquarantine.
- **Effect-handler runtime materialisation.** `!{sysreg}`, `!{ipc}` stay checked-not-enforced at emit.
- **Generic walker-chain activation for the post-PA7 surface.** Generics/traits/borrowed-refs stay side-table-level.
- **Multi-instruction sub-register sweeps beyond MOV.** V7 covers `mov` for sub-registers (`mov al, imm8`, `mov ax, imm16`, etc.). `add al, imm8` and other sub-register arithmetic forms are not in scope; the quarantined files do not use them.
- **`mov` between memory operands.** x86 forbids it without scratch; not in scope (no quarantined file needs `mov [a], [b]`).

This round is the smallest disciplined sequence that drains the 9-file checkpoint-2 quarantine, produces a kernel.elf whose codegen is correct (not silently 8x-wider or pointing every symbol at offset 0), and reopens R-phase work with full substrate.

### m1-002. cmd_build: thread `function_offsets` into per-symbol `st_value` via the PA7C-m1-001 symbol walk (V8)

- **Summary:** PA7C-m1-001 made every top-level `let NAME : T = fn (...)` produce a `SymbolEntry { name: "NAME", kind: SymbolKind::Function }` with the binding's real name in the ELF symbol table. PA7C-m2-001 made the `EmitWalker::visit_lambda` arm populate `function_offsets: HashMap<u32, u32>` (NodeId → byte offset in `.text`) at `emit_walker.rs:633`. PA7C-m1-003 unified byte tracking. **What is still missing:** `cmd_build.rs:767..783` reads `function_offsets` to compute per-symbol *size* (by sorting offsets and taking the next-offset delta) but never writes the offset itself into the `SymbolEntry::st_value` field. Every exported symbol still has `st_value: 0`. Linkers then resolve every cross-file `call NAME` to offset 0 of `.text` of the defining object — masked today only because (a) PLT32 relocs resolve to *runtime* addresses, not `st_value`; (b) the kernel doesn't currently call most symbols. The fix is a single read-write pair: where `cmd_build` constructs each `SymbolEntry` from a `function_offsets` entry, set `st_value = offset_in_text` and `st_size = next_offset - offset_in_text` (the size-computation logic already exists).
- **Acceptance criteria:**
  - Each `STT_FUNC` `SymbolEntry` emitted by `cmd_build` has `st_value = function_offsets[node_id]` and `st_size = (next_function_offset OR text_end) - st_value`.
  - The PA7C-m1-002 emitter assertion (`SymbolLayoutInvalid` for overlapping or out-of-bounds ranges) continues to pass after the change (verified by running the existing tests under `crates/paideia-as-emitter-elf/src/tests/symbol_layout`).
  - New integration test `crates/paideia-as/tests/build_emit_pa8_st_value.rs`: builds a 3-function source (`fn_a`, `fn_b`, `fn_c` each with a distinct trivial body shape), inspects the resulting `.o` via the `object` crate, asserts that `fn_b.st_value > fn_a.st_value`, `fn_c.st_value > fn_b.st_value`, and `fn_a.st_value + fn_a.st_size == fn_b.st_value` (contiguous packing).

### m2-001. emit_block_body: accept Branch as the final expression of unit-typed blocks (V2)

- **Summary:** PA7C-m3-003 (`592a40a`) made statement-position blocks accept a trailing `;` by introducing `BlockKind::Statement` + synthesising a final `IrKind::Unit` node. **The fix landed in the parser, but emit_walker's `emit_block_body` was not updated to handle the case where the *final expression* of a unit-typed block is itself a `Branch` (if-expression) whose arms have unit type.** Concretely, `slab_alloc` in `cap/slab.pdx:95..103` has the shape `fn () -> { if free_head >= CAP_SLAB_CAPACITY { CAP_SLAB_CAPACITY } else { let idx : u64 = free_head; free_head = free_list[free_head]; idx } }` — the final expression is a `Branch` whose then-arm is a literal and whose else-arm is a block ending in `idx`. emit_block_body's `IrKind::Branch` arm exists for value-position branches but does not handle this nested final-expression case. Fix: in `emit_block_body`, when the last `NodeId` in the block is `IrKind::Branch`, recursively call `emit_block_body` on each arm with the same `BlockKind` as the enclosing block, then emit a per-arm jump-to-end stitching pass at the end. The arms' tail expressions are the block's tail expression.
- **Acceptance criteria:**
  - `emit_block_body` recognises `IrKind::Branch` as the last element of a `Block` and dispatches per-arm.
  - For each arm, the arm body is elaborated; the arm's tail expression is the block's tail; the arm-end jumps are stitched to a single block-end label.

### m3-003. encoder: sub-register encoder width-awareness for the MOV family (V7)

### m5-002. encoder + IR: general memory operand `mov [base + disp]` for both load and store (V11)

Channel/IPC files need this for descriptor field reads (`(*ch).head` load) and writes (PA7C-m5-002 covered the write side via `IrKind::Store`, but the load side does not have a parallel IR shape — `IrKind::Load` either doesn't exist or carries only constant addresses). - **Acceptance criteria:**

  - Width variants (u8/u16/u32 destination) compose with V7's sub-register awareness.
  - Negative-displacement disp8 (`mov rax, [rdi - 8]`) is encoded correctly (`48 8B 47 F8`).
