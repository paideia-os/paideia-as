## 1. Tempo + cross-repo handoff

v0.9 runs continuously across all selected issues with **three named pause-and-verify checkpoints (1–3)** plus a round-close ritual. The checkpoint count drops from v0.8's five because v0.9's substrate work is narrower (3 gaps, not 11), and the cross-repo work fans out wider (rewrites + unquarantine + backlog resume) but consolidates into fewer cut-points.

Per memories `feedback_cross_repo_escalation`, `feedback_phase6_to_paideia_os_resume`, `feedback_paideia_as_version_discipline`, `feedback_paideia_os_tempo`, and `feedback_autonomous_tempo`: the checkpoint pauses are existing protocol. v0.9 is novel in that two of the three checkpoints span *both* repos in the same cut (checkpoint 2 unquarantines as a batch; checkpoint 3 resumes paideia-os-side autonomous work for two milestones in parallel).

### 1.1 Per-issue cycle (between checkpoints)

Per the user's directive — "softarch → workerbee → debugger per issue":

```
softarch  → produces the issue body (v0.9 template, §5)
              + commits to the GitHub issue (on the correct repo per §4.1)
              + dispatches workerbee
workerbee → implements the issue
              + runs cargo test --workspace      (paideia-as side)
              + runs ./tools/build.sh            (paideia-os side)
              + runs the cross-repo canaries (§6.1)
              + commits + pushes if green and (paideia-as) test count grew
debugger  → invoked ONLY when:
              (a) cargo red OR build.sh red, OR
              (b) test count flat or dropped (paideia-as side), OR
              (c) SARIF golden mismatch after intentional catalog change, OR
              (d) cross-repo canary red
            (no debugger invocation on green issues)

if green AND (paideia-as side) test count strictly grew AND canaries green:
  commit + push to main         (PaideiaOS-mode no-PR workflow, per Phase 4 §3)
  gh issue close <n>
  → immediately pick up next issue; no pause unless checkpoint-triggered
```

The three-agent rotation is unchanged from v0.8. v0.9 adds one nuance: paideia-os-side issues (the 5 rewrites + the unquarantine batch + the R6.5/D7 backlog issues) do **not** demand a test-count growth — paideia-os has no Rust test suite per memory `feedback_paideia_os_no_cicd`. Their cargo-green gate is `./tools/build.sh` exit 0 + non-empty `build/kernel.elf` + pre-push hook green.

### 1.2 Checkpoint 1 — paideia-as substrate ready

**Trigger:** all 3 paideia-as substrate issues (PA9-m1-001 P0158 elaboration, PA9-m1-002 P0211 elaboration, PA9-m1-003 P3-m2-003 encoder) closed.

**Rationale:** these three gaps are the *only* paideia-as-side prerequisites to the rest of v0.9. The 5 paideia-os pseudocode rewrites and the unquarantine batch both need the substrate to build the rewritten + unquarantined files. Bumping the submodule pin before all three close would either ship a broken pin or trigger duplicate rebumps — neither acceptable.

**Verification ritual:**

```bash
# In paideia-as (no-op; the closes already proved green):
cargo test --workspace
# Verify the three new regression tests name the specific cases they cover:
cargo test --workspace -- regression::p0158_bare_if_unit_return
cargo test --workspace -- regression::p0211_nested_array_repeat
cargo test --workspace -- regression::p3_m2_003_base_index_addressing

# In paideia-os (the cross-repo bump):
cd /home/snunez/Development/PaideiaOS
git submodule update --remote tools/paideia-as
./tools/build.sh

# Build must remain green at the existing unquarantined surface
# (channel.pdx / slab.pdx etc. are still quarantined here; only the
# already-on-main set must compile under the new pin).
test -s build/kernel.elf || { echo "FAIL: kernel.elf empty after submodule bump"; exit 1; }

# Commit the submodule bump.
git add tools/paideia-as
git commit -m "$(cat <<'EOF'
Bump paideia-as submodule (v0.9 checkpoint 1 — substrate gaps closed)

Closes PA9-m1-001 (P0158), PA9-m1-002 (P0211), PA9-m1-003 (P3-m2-003).
Build verified: ./tools/build.sh produces non-empty kernel.elf cleanly
against the new pin; the already-unquarantined surface still resolves.

Co-Authored-By: workerbee <noreply@anthropic.com>
EOF
)"
git push origin main
```

If `./tools/build.sh` is red after the bump, the failure mode is a *post-substrate regression in one of the three gap fixes themselves*. File a new paideia-as issue under `pa9-substrate`, label `regression-verify`, and resume the paideia-as autonomous loop on it. Do not bump the submodule pin yet; the bump commit only lands when the verification is green.

### 1.3 Checkpoint 2 — 5 rewrites + unquarantine batch

**Rationale:** the 5 rewrites convert pseudocode-stub `.pdx` files into real `.pdx` modules; the unquarantine batch moves the remaining 7 quarantined files (the 5 rewritten + `core/cap/slab.pdx` + `core/ipc/channel.pdx` + `core/sched/enqueue.pdx`; `tlb_shootdown.pdx` is already unquarantined at PaideiaOS commit `a682707`) back to `src/kernel/`. The single-batch shape (one issue covering all 7 file moves) keeps the commit history tight and is appropriate because every file is gated on the same paideia-as v0.9 substrate pin — there is no fan-out reason to fragment.

**Verification ritual:**

```bash

#   - moves the 7-file set in one git commit if rewrites landed under
#     .quarantine/ to keep the build canary clean during rewrite.
# osarch companion specifies which of the two shapes the rewrites take.

  - src/kernel/core/ipc/channel.pdx
```

If any of the 7 files trips a previously-unknown paideia-as gap at this point, run §9 cross-repo escalation: file a paideia-as issue under `pa9-substrate`, run the three-agent loop, close, bump submodule again, re-attempt the unquarantine for that file only. Per §8.4 rollback: do not leave a half-unquarantined state.

If `./tools/run-smoke.sh` is red — even with both backlogs closed — that is a P0 smell against one of the R6.5/D7 closures. Bisect against the last known-green smoke (the checkpoint-2 head SHA), file the regressing issue under either `r6.5-resume` or `d7-resume` per the affected surface, run the three-agent loop on it, re-verify, then proceed to round-close.

The substrate now has zero quarantine debt — the verb shift v0.9 owns.

|---|---|---|

Three intermediate checkpoints — one per cross-repo coupling cut. Each pause is bounded and short; the autonomous loop resumes the next checkpoint's work immediately on green verification.

## 2. PR sizing discipline

v0.9 keeps the v0.8 size bands but **forbids L this round**. The paideia-as substrate work is intentionally narrow (3 single-gap issues); the paideia-os rewrite work is 5 small files (~50–150 LOC each); the unquarantine batch is mechanical (git mv + commit). No issue should approach the M ceiling, and L is reserved for catastrophes that should not occur this round.

### 2.1 Size bands (this round)

| Size | Net diff (LOC, ex. generated, ex. corpus, ex. snapshots) | Examples | Files touched | Review target |
|---|---|---|---|---|
| **XS** | < 50 | PA9-m1-001 (P0158 elaboration patch + regression test), unquarantine batch (mechanical mv + 1 commit), R6.5/D7 single-issue closes that are review-only follow-ups | ≤ 3 | ≤ 10 min |
| **S** | 50–200 | PA9-m1-002 (P0211 elaborator extension + fixtures), PA9-m1-003 (P3-m2-003 base+index encoder + fixtures), most pseudocode rewrites (slots, allocator, mpsc_lock, destroy_channel) | ≤ 6 | ≤ 25 min |
| **M** | 200–400 | The dispatch rewrite if it grows the IPC op-code surface; any R6.5/D7 backlog issue that touched the IPC layer pre-pause | ≤ 12 | ≤ 45 min |
| **L** | 400–800 | **FORBIDDEN this round.** Any issue projecting above 400 LOC must split per §2.3. | n/a | n/a |

Generated code, test corpora, `Cargo.lock` lines, and SARIF snapshot regen output are excluded from the LOC count. The `.pdx` rewrites count source lines only (not the comment blocks they replace).

### 2.2 Pseudocode-rewrite sizing rule

The 5 paideia-os pseudocode rewrites are intentionally small:

| File | Pseudocode LOC (current) | Expected real-`.pdx` LOC | Band |
|---|---|---|---|
| `slots.pdx` | ~40 | ~60 | S |
| `allocator.pdx` | ~25 | ~50 | S |
| `dispatch.pdx` | ~30 | ~80 | S |
| `mpsc_lock.pdx` | ~35 | ~70 | S |
| `destroy_channel.pdx` | ~25 | ~50 | S |

If any rewrite projects above 200 LOC in real `.pdx`, the issue splits into "scaffold module" + "fill module" sub-issues. The hard split is grep-able from the `Size` line in the closing commit message (§5.1).

### 2.3 Hard-split triggers (this round)

A single v0.9 issue must split if any of:

1. Net diff projects above 400 LOC (always — L is forbidden).
2. It mixes a paideia-as substrate fix with a paideia-os rewrite (cross-repo work always lands as separate issues per repo).
3. It mixes two of the three substrate gaps (P0158, P0211, P3-m2-003) — each gap is its own observable test surface and each gets its own commit.
4. It mixes a rewrite with the unquarantine batch (rewrites land first; batch lands second).
5. It mixes an R6.5-resume issue with a D7-resume issue (different driver-vs-IRQ surfaces; different milestone).

### 2.4 No-PR mode reminder

Per Phase 4 retrospective §3 (carried through Phase 5, Phase 6, PA7-completion, v0.8): under PaideiaOS-mode, work lands as direct pushes to `main` after the cargo-green / build-green gate. The size bands still apply — the "PR" is the squash-merge commit. A single commit exceeding the band ceiling is forbidden the same way an XL PR would be in PR mode.

## 4. Milestone shape

v0.9 introduces **per-side milestones** — paideia-as carries 2 milestones for its narrow substrate scope; paideia-os carries 4 milestones for its broader work (rewrites + unquarantine + R6.5-resume + D7-resume). This is a refinement of v0.8's single-repo-five-milestone shape, because v0.9's work intentionally splits across both repos.

### 4.1 Naming + role

| Milestone | Repo | Title | Owns issues | Close trigger |
|---|---|---|---|---|
| `pa9-substrate` | paideia-as | v0.9 — Substrate gap fixes (P0158 + P0211 + P3-m2-003) | PA9-m1-001, PA9-m1-002, PA9-m1-003 | all 3 closed + checkpoint 1 verified (§1.2) |
| `pa9-closure` | paideia-as | v0.9 — Closure (version bump + CHANGELOG + tag) | (no issues; ritual-only) | §10.3 closure ritual complete |
| `r7-rewrite-checkpoint-2` | paideia-os | v0.9 R7 — Pseudocode rewrites (5 files) | R7-rewrite-001..005 | all 5 closed + reach checkpoint 2 readiness |
| `r7-unquarantine` | paideia-os | v0.9 R7 — Unquarantine batch (7 files) | R7-unquarantine-001 (batch) | closed + checkpoint 2 verified (§1.3) |
| `r6.5-resume` | paideia-os | v0.9 R6.5 — IRQ backlog resume | R6.5-resume-NNN (osarch enumerates) | all closed + checkpoint 3 part-1 (§1.4) |
| `d7-resume` | paideia-os | v0.9 D7 — Driver backlog resume | D7-resume-NNN (osarch enumerates) | all closed + checkpoint 3 part-2 (§1.4) |

**Total milestones: 6** (2 on paideia-as, 4 on paideia-os).

`pa9-substrate` gates checkpoint 1 (cross-repo submodule bump). `r7-rewrite-checkpoint-2` + `r7-unquarantine` jointly gate checkpoint 2 (cross-repo unquarantine batch). `r6.5-resume` + `d7-resume` jointly gate checkpoint 3 (paideia-os smoke green). `pa9-closure` is the bookend ritual.

### 4.2 Within-milestone sequencing

osarch owns the within-milestone order. The softarch invariants the order must respect:

1. **`pa9-substrate`**: PA9-m1-001 (P0158, simplest, smallest fixture) before PA9-m1-002 (P0211, mid-complexity) before PA9-m1-003 (P3-m2-003 encoder, largest). The order is "easiest first" so a regression in the harder issue does not block verification of the easier ones.
2. **`r7-rewrite-checkpoint-2`**: rewrites may land in any order, but `slots.pdx` first (the smallest, simplest, most isolated) so the rewrite *shape* gets one observable validation before the other four follow. The other four follow in size order ascending.
3. **`r7-unquarantine`**: single issue, single batch commit (§8.3).
4. **`r6.5-resume`**: osarch enumerates the prior R6.5 issue list; preserve the pre-pause order.
5. **`d7-resume`**: osarch enumerates the prior D7 issue list; preserve the pre-pause order. R6.5-resume and D7-resume work *interleave* per the existing standing pattern (one R6.5 issue, one D7 issue, alternating).
6. **`pa9-closure`**: no issues; only the ritual.

### 4.3 Round-close criterion (the bounded stop point)

The autonomous loop stops *only* when all of:

| # | Criterion | How verified |
|---|---|---|
