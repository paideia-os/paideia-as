## 0. Scope decision

Three scope options were considered:

- **(A) Minimum — three paideia-as gaps + five paideia-os rewrites + one batch unquarantine.** Closes the three encoder/elaborator blockers (P0158 bare-if, P0211 nested array-repeat, encoder base+index+scale+disp SIB form) and rewrites the five pseudo-assembly IPC stubs (slots, allocator, dispatch, mpsc_lock, destroy_channel) into real `.pdx` form. Total **8 issues** (3 paideia-as + 5 paideia-os) plus one batch unquarantine + one round-close. **Picked.**
- **(B) Extended — A + anticipated-gap sweeps.** Adds: the 9 deferred Mov-site sweep from PA8-m3-001 (sites 4..12 of the eleven Mov peers), a follow-up narrow-form Mov for r16-imm encoding, the still-deferred supervisor-mnemonic bridge to the unsafe-walker dispatch arms not covered by PA8-m5-001, and a peephole pass for the redundant `mov rax, rax` sequences emitted in `tlb_shootdown.pdx` as MMIO placeholders. Round size ~18–22 issues. Rejected: each of these is a *latent* concern, not a *blocker* of any file in `.quarantine/`. The v0.8 round's correctness-sweep discipline was justified because slab/IPC/sched files actively miscompiled under the v0.7.0 encoder; the v0.9 latent items do not block any unquarantined file's codegen. Bundling them costs ~12 issues and at least one extra week of paideia-as work to gate the R6.5 / D7 resume that is the actual user priority per [[feedback_paideia_os_tempo]].
- **(C) Full — A + B + Phase 4 surface adoption (paideia-as 0.9 surface: string literals, multiboot2 / PVH ELF Note, `else if` chain shortcut sugar, `for x in 0..N` desugaring to a `while`-loop).** Round size ~30+ issues. Rejected for v0.9 because the surface items are PaideiaOS quality-of-life rather than blockers; none of the 8 quarantined files use them; and v0.8 already deferred the equivalent items (V14 / V15 in the v0.8 plan) on the same grounds.

**Decision: option (A).** Justification:

1. **(A) drains the entire `.quarantine/` directory.** The diagnosis confirms (against actual `check` / `build` runs on each file at v0.8.0 HEAD) that exactly three paideia-as gaps and five rewrites cover all 8 remaining files. There is no fourth elaborator/encoder blocker hiding in the set.
2. **The five rewrites are not "deferred work" — they are the right correctness boundary.** Each of slots / allocator / dispatch / mpsc_lock / destroy_channel is a constants-only file (no functions, no IR-side codegen) that was checked into the kernel tree as pseudocode during early Phase 3 because the paideia-as 0.4 surface could not parse the intended form. The v0.8.0 surface now parses real `module = structure { let NAME : TYPE = VALUE }`. Rewriting them on the paideia-os side is correct hygiene; tracking them as paideia-as feature requests would be wrong.
3. **The three paideia-as gaps decouple cleanly.** PA9-m1-001 (bare if) touches `crates/paideia-as-parser/src/parse_control.rs` + `crates/paideia-as-elaborator/src/check_expr.rs`; PA9-m1-002 (nested array-repeat) touches `crates/paideia-as-elaborator/src/lower.rs` and reuses the V3/V13 machinery PA8-m2-002 landed; PA9-m1-003 (encoder SIB) touches `crates/paideia-as-encoder/src/encode_instruction.rs` only. No crate-boundary work, no schema migration, no surface-level rethink.
4. **The PaideiaOS rewrites are XS each.** Per the diagnosis, each pseudocode stub contains 2–4 integer constants and a comment block. The rewrite is mechanical: change `module Foo` + bare `NAME = VALUE` to `module Foo = structure { let NAME : u64 = VALUE }`. No fn bodies, no IR-emitting work.
5. **Option (B) double-pays the cross-repo coordination cost.** The user memory [[feedback_paideia_os_tempo]] is explicit that paideia-os runs continuously; per [[feedback_phase6_to_paideia_os_resume]] each paideia-as version close is followed immediately by a submodule bump + paideia-os resume. Bundling latent-correctness work into v0.9 would extend the paideia-as round by ~1.5–2 weeks and delay R6.5 / D7 by the same; the latent work can land in v1.0 alongside more surface work.

**Rejected alternative:** the four-gap split where rewrites go on a separate paideia-os feature branch and merge after v0.9 closes. Rejected because that scheme creates two unquarantine sweeps in the same calendar window: one for the 3 real-code files (slab, channel, enqueue) post-PA9-m1, then another for the 5 rewritten stubs. Bundling them into one m6 batch unquarantine is cleaner and matches the v0.8 m7-002 pattern.

### PA9-m1-001. parser + elaborator: accept bare `if EXPR { ... }` as a unit-typed statement (no else required)

- **Summary:** `core/cap/slab.pdx:111..116` writes `let slab_free : (u64) -> () = fn (idx : u64) -> { if idx < CAP_SLAB_CAPACITY { free_list[idx] = free_head; free_head = idx; } }`. The fn body is a block whose tail expression is the `if`. The `if` has no `else` arm. The then-block contains two assignments terminated by `;` and no tail expression. Today, `parse_expr` dispatches `if` via `parse_if(BlockKind::Value)` (at `parse_expr.rs:72`), which routes the then-block through `parse_block_kind(BlockKind::Value)`. That, in turn, fires P0158 (`block expression must have a final expression; trailing ';' is not allowed`) because the then-block ends in `;` with no tail. **Fix:** when `parse_if` discovers no `else` keyword after the then-block (i.e., `else_block` would be `None`), the entire if-expression is unit-typed and the then-block should be parsed in `BlockKind::Statement` instead of `BlockKind::Value`. The change is localised to `parse_if`: parse the then-block with `BlockKind::Statement` first (which accepts trailing `;` and synthesises `()` as the tail), then check for `else`; if `else` is present, *re-parse* the then-block with the requested kind (or, simpler: speculatively descend into the if-token, peek past the then-block for `else`, and dispatch the BlockKind accordingly). The simpler form: in `parse_if`, after parsing the then-block in `BlockKind::Statement`, when no `else` follows, synthesise an empty `else_block` containing a unit literal; the elaborator already knows how to type-check unit branches.
- **Acceptance criteria:**
  - `parse_if(BlockKind::Value)` continues to require both arms when `else` is present (existing tests at `crates/paideia-as-parser/src/parse_control.rs:849..914` continue to pass — verify by running `cargo test -p paideia-as-parser parse_if`).
  - `parse_if(BlockKind::Value)` with no `else` parses the then-block in `BlockKind::Statement` and produces an `ExprIf` whose `else_block: None` is implicitly unit. The elaborator's check_expr arm for `ExprIf` infers unit type when `else_block.is_none()` and applies the existing unit-coercion logic to the then-block.
  - The elaborator's check_expr arm for `ExprIf` continues to error P0158 (or a new P0159 if the diagnostic catalog wants the distinction) when the surrounding context demands a non-unit value AND `else_block.is_none()`. I.e., `let x : u64 = if cond { 1 }` (no else, value position) fails with a clear diagnostic naming the missing else arm.
  - **Fixture:** `tests/build-emit/pa9_bare_if_unit.pdx` containing the exact `slab_free` shape: `module M = structure { let mut head : u64 = 0; let mut arr : [u64; 4] = [0, 0, 0, 0]; let f : (u64) -> () = fn (idx : u64) -> { if idx < 4 { arr[idx] = head; head = idx; } } }`. Builds + encodes; the resulting `.o`'s `.text` contains a `cmp` + `jcc` + assignment sequence + `ret`.
  - **Negative fixture:** `tests/build-emit/pa9_bare_if_value_position.pdx` containing `let x : u64 = if cond { 1 }` (no else) fails with diagnostic P0158 or P0159 naming the missing else arm. Build fails cleanly with a non-zero exit and a SARIF entry.
  - 4 unit tests in `crates/paideia-as-parser/src/parse_control.rs` test module: (a) bare-if in stmt-position parses, (b) bare-if in fn-body-as-tail parses, (c) bare-if + else in stmt-position parses, (d) bare-if + else in value-position parses; plus 2 elaborator tests in `crates/paideia-as-elaborator/src/check_expr.rs` test module: (e) bare-if in unit fn-body infers unit, (f) bare-if in value position errors.
  - The catch-all P0158 emission at `parse_control.rs:329..339` is *not* removed — it still applies to non-if blocks ending in `;`. Only the if-specific path is rerouted.
- **Files:** `crates/paideia-as-parser/src/parse_control.rs` (the `parse_if` function; the `parse_block_kind` arm-routing change), `crates/paideia-as-elaborator/src/check_expr.rs` (the `ExprIf` arm — confirm/extend unit-coercion when `else_block.is_none()`), `tests/build-emit/pa9_bare_if_unit.pdx` (new), `tests/build-emit/pa9_bare_if_value_position.pdx` (new), 6 test files (4 parser, 2 elaborator).

### PA9-m1-002. elaborator: materialise nested array-repeat `[expr; N]` inside struct field initialisers (P0211)

- **Summary:** `core/ipc/channel.pdx:58..93` declares `let mut channel_pools : [[Channel; 32]; 1] = [ [ Channel { producer_cap: 0, consumer_cap: 0, head: 0, tail: 0, slots: [Message { length: 0, payload: [0; 256] }; 64] }; Channel { ... }; ... ] ]`. The outer array is a 1-element 2D shape `[[Channel; 32]; 1]`. The inner array is 32 enumerated Channel literals separated by `;`. Each Channel literal has a `slots:` field initialised with `[Message { length: 0, payload: [0; 256] }; 64]` — a 64-element array-repeat whose element is a Message record literal whose `payload:` field is itself a 256-element array-repeat of byte zeros. Two nested array-repeat layers + a record literal at the inner level. PA8-m2-002 landed the simple shape `[expr; N]` for *flat* element types (integer literals at module level). PA8-m2-003 landed module-level record-literal lowering. **The gap:** the elaborator's `lower.rs` arm for `ExprData::ArrayRepeat` does not recurse into element expressions when the element is itself a `RecordLit` whose fields contain `ArrayRepeat`. The encoder dispatch then hits P0211 (`array repeat count must be constant literal integer`) on the inner `[0; 256]` because the outer ArrayRepeat's lowering computed a partial materialisation that left the inner ArrayRepeat unresolved. **Fix:** extend the `ExprData::ArrayRepeat` lowering arm to be fully recursive: when the element expression is itself an `ArrayRepeat` or contains a `RecordLit` whose fields contain `ArrayRepeat`, recursively materialise the element bytes before replicating N times. Equivalent: a single elaboration pass that walks the AST bottom-up materialising every constant ArrayRepeat into a flat `bytes: Vec<u8>` blob, then composing the blobs at the higher levels.
- **Acceptance criteria:**
  - The elaborator pass `materialise_constant_array_repeats` runs as part of the `lower.rs` module-level initialiser handling; it visits every `ArrayRepeat` and `RecordLit` reachable from a module-level `let` RHS and produces a flat `bytes: Vec<u8>` for each constant subtree.
  - The pass is bottom-up: nested `ArrayRepeat` inside `RecordLit` field initialisers materialise first, then the outer `RecordLit` composes the field bytes per `RecordLayoutTable` (PA8-m2-003 / Phase 6 m1 infrastructure), then the outer `ArrayRepeat` replicates the composed record bytes N times.
  - The pass refuses non-constant expressions in module-level initialisers with a clear diagnostic (T0541 already exists from PA8-m2-003; reuse it for the bottom-up case).
  - **Fixture:** `tests/build-emit/pa9_nested_array_repeat.pdx` with a simplified Channel struct (3 u64 fields plus a `payload: [u8; 8]` array field) and `let mut pools : [[Channel; 4]; 2] = [ [ Channel { a: 0, b: 0, c: 0, payload: [0; 8] }; 4 ]; 2 ]`. Builds; `readelf -x .data <obj>` shows the bytes packed as 2 × 4 × (24 + 8) = 256 bytes of zeros in the expected layout.
  - **Larger fixture:** `tests/build-emit/pa9_channel_pools_subset.pdx` reproducing the channel.pdx shape but with reduced dimensions (`[[Channel; 2]; 1]` instead of `[[Channel; 32]; 1]`, `slots: [Message; 4]` instead of 64, `payload: [u8; 16]` instead of 256) — verifies the full nesting depth (array-of-array-of-record-with-record-array-with-byte-array) end-to-end.
  - **Diagnostic preservation:** P0211 continues to fire on the *real* error case (non-constant repeat count: `let arr : [u64; n] = [0; n]` where `n` is a runtime variable). New negative test asserts this.
  - 5 unit tests in `crates/paideia-as-elaborator/tests/lower/nested_array_repeat.rs`: (a) flat array of ints `[0; 8]`, (b) nested array of ints `[[0; 4]; 2]`, (c) array of records `[Rec { x: 0 }; 4]`, (d) array of records-with-array-field `[Rec { x: 0, ys: [0; 4] }; 2]`, (e) two-level nested array of records-with-array-field (the channel_pools shape).
- **Files:** `crates/paideia-as-elaborator/src/lower.rs` (the `ExprData::ArrayRepeat` arm; the new `materialise_constant_array_repeats` pass), `tests/build-emit/pa9_nested_array_repeat.pdx` (new), `tests/build-emit/pa9_channel_pools_subset.pdx` (new), 5 test files.

### PA9-m1-003. encoder: extend `mov` dispatch to the general base+index*scale+disp SIB form

- **Summary:** `core/sched/enqueue.pdx:60` reads `let head : u64 = heads[priority];` where `heads : [u64; LEVELS]` is a module-level array and `priority` is a u64 fn parameter. The elaborator lowers the access to an IR shape with operand `Operand::MemSib { base: <heads_base>, index: Some(<priority_reg>), scale: Scale::X8, disp: 0 }`. The encoder's current `encode_mov` dispatch (at `crates/paideia-as-encoder/src/encode_instruction.rs:507..526`) has an arm for `[Operand::Reg(dest), Operand::MemSib { base, index: Some(index), scale, disp: 0 }]` — but `enqueue.pdx:63` writes `heads[priority] = tcb;`, which lowers to `[Operand::MemSib { base, index: Some(index), scale, disp: 0 }, Operand::Reg(src)]` — the *write* direction of the same SIB shape. The current dispatch table has no arm for the write direction; it falls through to the catch-all at line 579 and emits `"mov form not in phase-3-m2-002 minimum"`. **Additionally:** any access with a non-zero displacement (e.g., `heads[priority] = tcb + 8` would lower to disp=8 if elaborator decides to bake the offset into the address — current behavior loads the constant into a reg first, but future elaboration paths may emit MemSib with disp != 0) hits the same gap. **Fix:** add two new arms to `encode_mov`: (a) write side of base+index*scale+disp=0: `[MemSib { base, index: Some, scale, disp: 0 }, Reg]`; (b) read side of base+index*scale+disp=N: `[Reg, MemSib { base, index: Some, scale, disp: N }]`; (c) write side of base+index*scale+disp=N: `[MemSib { base, index: Some, scale, disp: N }, Reg]`. Three new arms total. Each emits a SIB byte (scale<<6 | (index_id&7)<<3 | (base_id&7)) and a disp32 (or disp8 for small N if the encoder wants the short form; pick disp32 for simplicity, leave the disp8 short-form as a future optimisation).
- **Acceptance criteria:**
